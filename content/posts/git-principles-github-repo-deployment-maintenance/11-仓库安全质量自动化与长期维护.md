# 仓库安全质量自动化与长期维护

### 最小权限原则

仓库权限应遵循最小权限原则。不是所有成员都需要主分支写权限，也不是所有机器人都需要管理员权限。

常见角色设计：

| 角色 | 权限 | 说明 |
| --- | --- | --- |
| Owner/Admin | 管理仓库设置和权限 | 少量负责人持有 |
| Maintainer | 合并 PR、管理 Release | 核心维护者 |
| Developer/Write | 推送功能分支、提交 PR | 普通开发者 |
| Triage | 管理 Issue 和 PR 标签 | 社区协作者 |
| Read | 只读访问 | 外部审计或观察者 |

组织仓库应优先通过团队授权，而不是逐个用户授权。成员离职或角色变化时，团队权限比个人权限更容易维护。

### 分支与标签保护

主分支保护防止未审查代码进入主线。标签保护防止发布版本被删除或移动。发布项目至少应保护：

```text
main
release/*
v*
```

推荐策略：

| 对象 | 规则 |
| --- | --- |
| `main` | PR + 审查 + CI + 禁止强推 |
| `release/*` | 限制发布负责人，禁止强推和删除 |
| `v*` | 仅维护者可创建，禁止更新和删除 |

如果平台支持 rulesets，可用规则集统一管理分支和标签策略，减少多个保护规则之间的优先级混乱。

### 密钥与敏感信息治理

密钥绝不能进入 Git 历史。`.gitignore` 只能防止未来误加文件，不能清除已经提交的敏感信息。常见敏感内容包括：

| 类型 | 示例 |
| --- | --- |
| API Token | `ghp_...`、云厂商 AK/SK |
| 私钥 | `id_rsa`、`.pem` |
| 数据库密码 | `.env`、配置文件 |
| 证书 | p12、jks、crt/key |
| 生产配置 | 真实域名、账号、内部地址 |

如果密钥已提交，应立即轮换密钥，然后清理历史。清理历史可用 `git filter-repo` 或 BFG Repo-Cleaner，但这会重写历史，需要团队统一停写、重新克隆或重置本地仓库。不要以为删除文件并提交就解决问题，因为旧提交仍包含敏感内容。

### 大文件与仓库体积

仓库体积膨胀会影响 clone、fetch、CI 和平台存储。常见原因：

| 原因 | 处理方式 |
| --- | --- |
| 提交构建产物 | 使用 `.gitignore` 忽略，制品放 Release 或包仓库 |
| 提交二进制大文件 | 使用 Git LFS 或对象存储 |
| 反复提交日志/数据集 | 改用外部存储，仓库只保存元数据 |
| 删除后仍占历史 | 使用历史清理工具并强制同步 |

查看对象大小：

```bash
git count-objects -vH
```

查找大对象示例：

```bash
git rev-list --objects --all \
  | git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' \
  | sort -k3 -n \
  | tail -20
```

### CI 质量门禁

建议把基础质量检查放到 PR 阶段：

| 检查 | 目的 |
| --- | --- |
| 格式化 | 避免风格争论 |
| Lint | 发现潜在错误和规范问题 |
| 单元测试 | 验证核心逻辑 |
| 集成测试 | 验证模块协作 |
| 构建 | 确认可交付 |
| 安全扫描 | 检查依赖漏洞和密钥 |
| 覆盖率 | 观察测试质量趋势 |

不要把所有检查都放到开发者本地自觉执行。关键检查应由 CI 强制执行，并与保护分支状态检查绑定。

### 提交签名与可追溯性

提交签名和标签签名用于证明提交或发布标签确实由可信身份创建。Git 支持 GPG、SSH 等签名方式。

创建签名提交：

```bash
git commit -S -m "fix: validate token issuer"
```

创建签名标签：

```bash
git tag -s v1.0.0 -m "Release v1.0.0"
```

签名不能替代代码审查，但能增强供应链可信度。对于开源库、CLI 工具、基础镜像、SDK 等被下游依赖的项目，建议要求发布标签签名。

### 归档与废弃仓库

仓库不再维护时，应明确状态，而不是让用户猜测。推荐步骤：

1. README 顶部添加维护状态说明。
2. 标记最后一个 Release。
3. 关闭或限制 Issue。
4. 如有替代项目，提供迁移链接。
5. 在平台上归档仓库。

归档前可以创建最终标签：

```bash
git tag -a v1.9.9 -m "Final maintenance release"
git push origin v1.9.9
```

### 备份与灾难恢复

远程平台不是唯一备份。重要仓库应至少有：

| 层次 | 方案 |
| --- | --- |
| Git 镜像 | `git clone --mirror` 定期备份 |
| 制品备份 | Release 附件、包、镜像单独备份 |
| 配置备份 | CI/CD、Secrets、环境变量记录在安全系统 |
| 权限备份 | 组织、团队、保护规则定期审计 |

镜像备份示例：

```bash
git clone --mirror git@github.com:org/demo-service.git
cd demo-service.git
git remote update --prune
```

恢复到新远程：

```bash
git push --mirror git@gitee.com:org/demo-service.git
```

`--mirror` 会覆盖目标端引用，恢复前必须确认目标仓库用途。

### 维护节奏

长期维护不是偶发任务，而应形成节奏：

| 周期 | 任务 |
| --- | --- |
| 每次 PR | 审查、CI、关联 Issue、更新文档 |
| 每周 | 清理过期分支、处理依赖更新 |
| 每月 | 审计权限、检查大文件、整理 Issue |
| 每个版本 | 更新 CHANGELOG、打标签、发布 Release |
| 每季度 | 复核分支策略、CI 性能、备份可用性 |

维护目标是让仓库始终处于可理解、可构建、可发布、可恢复的状态。
