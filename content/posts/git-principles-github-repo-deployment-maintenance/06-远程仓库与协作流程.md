# 远程仓库与协作流程

### remote 的本质

远程仓库配置保存在 `.git/config` 中。远程名通常为 `origin`，但这只是约定，不是强制规则。一个本地仓库可以配置多个远程，例如 `origin` 指向自己的 fork，`upstream` 指向上游开源项目。

查看远程：

```bash
git remote -v
```

添加远程：

```bash
git remote add origin git@github.com:org/demo-service.git
```

修改远程地址：

```bash
git remote set-url origin git@gitee.com:org/demo-service.git
```

删除远程：

```bash
git remote remove origin
```

查看远程详情：

```bash
git remote show origin
```

### fetch、pull 与 push

`fetch` 只从远程下载对象并更新远程跟踪分支，不会自动修改当前工作分支：

```bash
git fetch origin
```

`pull` 相当于 `fetch` 加整合操作。默认可能是 merge，也可能因配置变成 rebase：

```bash
git pull origin main
git pull --rebase origin main
```

`push` 将本地引用和对象发送到远程：

```bash
git push origin main
```

首次推送并建立上游关系：

```bash
git push -u origin main
```

建立上游后，可以直接：

```bash
git pull
git push
```

常用 push 参数：

| 参数 | 作用 |
| --- | --- |
| `-u` / `--set-upstream` | 设置当前分支的上游分支 |
| `--force-with-lease` | 较安全的强推，远程未被他人更新时才覆盖 |
| `--tags` | 推送所有标签 |
| `--delete` | 删除远程分支或标签 |
| `--dry-run` | 试运行，不实际推送 |

不要用 `--force` 覆盖共享分支。确实需要重写个人远程分支时，优先使用 `--force-with-lease`。

### 远程跟踪分支

执行 `git fetch origin` 后，本地会更新 `origin/main`、`origin/develop` 等远程跟踪分支。它们是本地缓存，不是可直接提交的普通本地分支。

基于远程分支创建本地分支：

```bash
git switch -c feature/login origin/main
```

或跟踪同名远程分支：

```bash
git switch --track origin/develop
```

查看本地分支和上游关系：

```bash
git branch -vv
```

清理远程已删除分支的本地缓存：

```bash
git fetch --prune
```

也可全局配置：

```bash
git config --global fetch.prune true
```

### Fork 工作流

开源项目常用 Fork 工作流。开发者没有上游仓库写权限时，先 fork 到个人空间，再把个人仓库克隆到本地。

推荐远程配置：

```bash
git remote -v
git remote add upstream git@github.com:upstream/demo-service.git
```

同步上游：

```bash
git fetch upstream
git switch main
git merge --ff-only upstream/main
git push origin main
```

开发功能：

```bash
git switch -c feature/login
git commit -m "feat: add login"
git push -u origin feature/login
```

随后在平台上从 `origin/feature/login` 向 `upstream/main` 创建 Pull Request。

### 同仓库分支工作流

企业内部项目常用同仓库分支工作流。成员对同一仓库有写权限，但主分支受保护，不能直接推送。

流程如下：

```bash
git fetch origin
git switch -c feature/payment origin/main
```

完成开发后：

```bash
git push -u origin feature/payment
```

在平台创建 Pull Request，目标分支为 `main`。通过代码审查、测试和状态检查后合并。合并后删除远程功能分支：

```bash
git push origin --delete feature/payment
```

本地清理：

```bash
git switch main
git pull --ff-only
git branch -d feature/payment
git fetch --prune
```

### 保护分支与协作治理

保护分支用于限制重要分支的直接推送、强制推送和删除，并要求 Pull Request 审查、状态检查、线性历史或签名提交等。保护分支不是 Git 协议本身的功能，而是托管平台在接收 push 或合并请求时执行的策略。

推荐至少保护以下分支：

| 分支模式 | 保护策略 |
| --- | --- |
| `main` | 禁止直接推送，要求 PR、至少一名审核、CI 通过 |
| `release/*` | 禁止强推和删除，限制可合并人员 |
| `hotfix/*` | 要求审查，但保留快速修复通道 |
| `docs/*` | 可放宽 CI 要求，但仍建议审查 |

一个成熟仓库的协作流程应把“谁能写”“谁能审”“什么条件能合并”“发布如何触发”全部写清楚，而不是依赖团队成员口头约定。

### 子模块与大文件

子模块适合把另一个 Git 仓库嵌入当前仓库：

```bash
git submodule add git@github.com:org/shared-lib.git libs/shared-lib
git submodule update --init --recursive
```

子模块记录的是外部仓库的某个提交 ID，而不是自动跟随外部仓库最新分支。因此更新子模块需要进入子模块目录拉取并提交新的指针。

大文件不适合直接提交到普通 Git 历史。二进制包、模型、视频、设计源文件应考虑 Git LFS 或制品仓库。直接提交大文件后，即使删除文件，历史中的对象仍会占用仓库体积，清理成本较高。
