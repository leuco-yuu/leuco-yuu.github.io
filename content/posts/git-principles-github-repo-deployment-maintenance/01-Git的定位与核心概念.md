# Git 的定位与核心概念

## Git 与 Git 管理区域

Git 是一个**分布式版本控制系统**。

- **“分布式”**：是指每个开发者本地克隆到的仓库都包含**完整的版本历史、对象数据库、分支引用和标签引用**。
- 与**集中式**版本控制相比：Git 的核心能力并不依赖远程平台。即使断开网络，开发者仍然可以**创建提交**、**切换分支**、**查看历史**、**比较差异**、**回滚版本**、**创建标签**，只有在需要与他人交换历史时，才需要执行 `fetch`、`pull`、`push` 等远程操作。

Git 的工作对象不是“单个文件的增量补丁”，而是项目在某一时间点的**快照**。每次提交都会记录一棵目录树，这棵树描述了该提交下所有被跟踪文件的路径、权限和内容对象。Git 会通过内容寻址复用未变化的对象，因此“快照模型”并不意味着每次提交都完整复制全部文件。`commit`、`branch`、`tag`、`merge`、`rebase` 都可以统一解释为对对象和引用的操作。

Git 日常使用中最容易混淆的是**工作区**、**暂存区**和**版本库**。

- **工作区**是实际编辑文件的目录。
  - 开发者在编辑器中创建、修改、删除文件，首先发生在工作区。
  - 工作区里的改动不一定会进入下一次提交，因为 Git 不会自动把所有修改纳入版本历史。

- **暂存区**也称 index，是下一次提交的候选集合。
  - 执行 `git add` 后，Git 会把指定文件当前版本写入暂存区。
  - 暂存区使开发者可以把一批工作拆分为多个有意义的提交。
  - 例如同一天修改了登录逻辑、文档和测试用例，可以只把登录逻辑相关文件加入暂存区并提交，再提交文档和测试。

- **版本库**是 `.git` 目录及其中的对象数据库、引用、配置和日志。
  - 执行 `git commit` 时，Git 会基于暂存区生成新的提交对象，并移动当前分支引用。

典型流程如下：

```bash
git status
git add src/auth/login.ts
git commit -m "feat(auth): add login token refresh"
```

这三条命令分别对应：

- “查看工作区与暂存区状态”；
- “把改动放入暂存区”；
- “把暂存区固化为提交”。

> [!ANNOT]
>
> 工作区指当前正在编辑的项目文件夹，暂存区指提交前的”待提交清单“，版本库指Git真正保存历史记录的地方。这三步都是在本地进行的，不会推送到远程仓库。

### Git 的状态语言

Git 使用**状态**描述文件是否被跟踪，以及改动处在哪个区域。

- `untracked` 表示文件存在于工作区，但尚未纳入 Git 管理。
- `modified` 表示已跟踪文件在工作区被修改，但修改尚未提交。
- `staged` 表示文件的某个版本已放入暂存区。
- `committed` 表示内容已存在于版本库对象数据库，并被某个提交引用。

使用 `git status -sb` 可以输出更适合脚本和日常快速查看的简短状态：

```bash
git status -sb
```

常见状态码含义如下：

| 状态 | 含义 |
| --- | --- |
| `??` | 未跟踪文件 |
| `M` | 已修改文件 |
| `A` | 新增到暂存区 |
| `D` | 删除文件 |
| `R` | 重命名文件 |
| `UU` | 合并冲突未解决 |

### Git 命令的设计层次

Git 命令可以分为高层命令和底层命令。日常开发主要使用高层命令，例如 `add`、`commit`、`branch`、`switch`、`merge`、`rebase`、`fetch`、`pull`、`push`。底层命令用于直接操作对象、引用或索引，例如 `hash-object`、`cat-file`、`update-index`、`write-tree`、`commit-tree`、`update-ref`。

理解底层命令不是为了在日常工作中频繁使用它们，而是为了建立正确心智模型。例如：

```bash
git cat-file -t HEAD
git cat-file -p HEAD
git cat-file -p HEAD^{tree}
```

第一条命令查看 `HEAD` 指向对象的类型，第二条命令打印提交对象内容，第三条命令查看提交所指向的目录树。通过这些命令可以看到，提交对象保存了树对象、父提交、作者、提交者和提交说明。

### Git 与代码托管平台的关系

GitHub、Gitee、GitLab、Bitbucket 等平台不是 Git 本身，而是围绕 Git 仓库提供远程托管、权限控制、代码审查、Issue、Wiki、CI/CD、Release、Pages、组织管理、安全扫描等协作能力的平台。Git 解决的是版本历史与对象交换问题；平台解决的是多人协作、权限治理、项目管理和交付发布问题。

因此，学习 Git 时应区分三类能力：

| 层次 | 典型内容 | 是否依赖平台 |
| --- | --- | --- |
| 本地版本控制 | `init`、`add`、`commit`、`branch`、`merge`、`rebase` | 否 |
| 远程同步 | `remote`、`fetch`、`pull`、`push` | 需要远程仓库，但不限平台 |
| 平台协作 | Pull Request、Issue、Release、保护分支、Actions、Pages | 是 |

这一区分非常重要。很多看似 Git 的问题，例如“为什么不能直接推送到 main”“为什么合并需要审批”“为什么 Release 页面没有构建产物”，实际是平台策略、权限规则或 CI/CD 配置问题，而不是 Git 命令本身的问题。

### Git 学习主线

建议按照以下主线学习 Git：

先理解对象模型。提交不是简单文本日志，而是带父子关系的对象图；分支不是副本，而是一个可移动引用；标签不是分支，而是一个通常不移动的版本标记。

再掌握本地提交。包括工作区、暂存区、提交、撤销、比较、历史查看、忽略文件和配置。

随后学习分支协作。重点是 `merge` 与 `rebase` 的差异、冲突解决、远程跟踪分支、分支命名与生命周期管理。

最后把 Git 放到平台中使用。重点是 GitHub/Gitee 仓库创建、权限配置、保护分支、Pull Request、镜像同步、Release、标签、制品上传和长期维护。
