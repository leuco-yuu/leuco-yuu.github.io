# 工作区暂存区提交与回退

### 查看状态与差异

`git status` 是最重要的日常命令。建议在执行提交、合并、变基、重置前都先查看状态：

```bash
git status
git status -sb
```

查看未暂存差异：

```bash
git diff
```

查看已暂存、即将进入下一次提交的差异：

```bash
git diff --staged
```

查看某个提交的变更：

```bash
git show --stat HEAD
git show --name-only HEAD
git show HEAD -- src/main.ts
```

常用 diff 参数如下：

| 参数 | 作用 |
| --- | --- |
| `--stat` | 显示文件级统计 |
| `--name-only` | 只显示文件名 |
| `--name-status` | 显示文件名和状态 |
| `--word-diff` | 按词显示差异 |
| `--cached` / `--staged` | 比较暂存区与 HEAD |

### 精准添加到暂存区

最常用的添加方式：

```bash
git add README.md
git add src tests
git add .
```

`git add .` 会添加当前目录及子目录中的改动，包括新文件和删除。`git add -A` 从仓库根视角添加所有改动。多数项目中二者效果接近，但在子目录执行时仍需理解路径范围。

交互式添加可以把一个文件的多处改动拆分到不同提交：

```bash
git add -p src/auth/login.ts
```

常见交互选项：

| 选项 | 含义 |
| --- | --- |
| `y` | 暂存当前 hunk |
| `n` | 不暂存当前 hunk |
| `s` | 拆分 hunk |
| `e` | 手动编辑 hunk |
| `q` | 退出 |

### 提交规范与参数

基础提交：

```bash
git commit -m "feat(auth): add password login"
```

常用参数如下：

| 参数 | 作用 |
| --- | --- |
| `-m` | 直接指定提交标题或说明 |
| `-a` | 自动暂存已跟踪文件的修改和删除，不包括新文件 |
| `--amend` | 修改上一次提交 |
| `--no-edit` | 保留原提交说明，常与 `--amend` 使用 |
| `--allow-empty` | 创建空提交，常用于触发 CI |
| `--author` | 指定作者信息 |
| `-S` | 使用 GPG/SSH 签名提交 |

修改最近一次提交：

```bash
git add README.md
git commit --amend
```

如果该提交已经推送到公共分支，修改提交会重写历史，需要谨慎。多人共享分支不建议 amend 后强推，除非团队明确采用该策略并开启保护措施。

推荐提交信息结构：

```text
<type>(<scope>): <subject>

<body>

<footer>
```

示例：

```text
feat(api): add user profile endpoint

Add GET /users/me endpoint and response DTO.

Refs: #128
```

常见类型包括 `feat`、`fix`、`docs`、`style`、`refactor`、`test`、`build`、`ci`、`chore`、`perf`。

### 删除、移动与重命名

删除已跟踪文件：

```bash
git rm old-config.yml
```

仅从版本控制中移除，保留工作区文件：

```bash
git rm --cached .env
```

移动或重命名：

```bash
git mv src/app.ts src/main.ts
```

Git 没有单独保存“重命名对象”，而是在比较历史时基于相似度检测重命名。因此手动 `mv` 后再 `git add -A`，通常也能被识别为重命名。

### 恢复与撤销

Git 新版本推荐使用 `restore` 和 `switch` 分担过去 `checkout` 的职责。

丢弃工作区某文件修改：

```bash
git restore src/main.ts
```

取消暂存，但保留工作区修改：

```bash
git restore --staged src/main.ts
```

从某个提交恢复文件：

```bash
git restore --source=HEAD~1 -- README.md
```

`reset` 用于移动分支引用并调整暂存区/工作区：

```bash
git reset --soft HEAD~1
git reset --mixed HEAD~1
git reset --hard HEAD~1
```

三种模式差异：

| 模式 | 移动 HEAD | 改暂存区 | 改工作区 | 典型用途 |
| --- | --- | --- | --- | --- |
| `--soft` | 是 | 否 | 否 | 撤销提交但保留暂存 |
| `--mixed` | 是 | 是 | 否 | 撤销提交并取消暂存 |
| `--hard` | 是 | 是 | 是 | 彻底回到指定提交 |

`--hard` 会丢弃工作区修改，执行前必须确认没有未备份的重要内容。

### revert 与 reset 的区别

`reset` 重写当前分支历史，适合本地未共享提交。`revert` 创建一个新的反向提交，适合撤销已经推送到共享分支的提交。

撤销某个提交：

```bash
git revert <commit>
```

撤销一个合并提交需要指定主线父提交：

```bash
git revert -m 1 <merge-commit>
```

`-m 1` 表示以第一个父提交为主线。错误选择主线会导致撤销方向错误，因此处理合并提交前应先查看：

```bash
git show --no-patch --pretty=raw <merge-commit>
```

### 临时保存工作进度

`stash` 适合临时切换任务：

```bash
git stash push -m "wip: debug login timeout"
git stash list
git stash show -p stash@{0}
git stash pop
```

常用参数：

| 参数 | 作用 |
| --- | --- |
| `-m` | 添加说明 |
| `-u` / `--include-untracked` | 同时保存未跟踪文件 |
| `-a` / `--all` | 同时保存被忽略文件 |
| `--keep-index` | 保留暂存区内容 |

`stash pop` 会应用并删除 stash；`stash apply` 只应用不删除。重要修改不应长期只放在 stash 中，应尽快提交到临时分支。
