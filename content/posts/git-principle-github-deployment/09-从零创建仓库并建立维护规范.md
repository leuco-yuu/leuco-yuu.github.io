# 从零创建仓库并建立维护规范

## 仓库命名与目录规划

创建仓库前，应先确定仓库边界。一个仓库应围绕一个清晰交付单元：一个服务、一个库、一个前端应用、一个文档站点，或一个 monorepo。仓库边界不清会导致权限、CI、发布和版本管理复杂化。

仓库命名建议：

| 规则 | 示例 |
| --- | --- |
| 使用小写和短横线 | `payment-service` |
| 避免无意义缩写 | `user-center` 优于 `uc` |
| 表明技术或领域 | `docs-platform`、`go-agent` |
| 避免环境名 | 不建议 `prod-api`、`test-web` |

推荐初始目录：

```text
demo-service/
  .github/ or .gitee/
  docs/
  src/
  tests/
  scripts/
  README.md
  CHANGELOG.md
  CONTRIBUTING.md
  LICENSE
  .gitignore
  .gitattributes
```

不同语言可调整结构，但 README、忽略规则、许可证、贡献指南、变更记录应尽早建立。

## 本地初始化完整示例

```bash
mkdir demo-service
cd demo-service
git init -b main

mkdir -p src tests docs scripts
cat > README.md <<'EOF'
Demo Service

Demo Service is an example backend service used to demonstrate repository governance.

### Quick start

npm install
npm test
EOF

cat > .gitignore <<'EOF'
node_modules/
dist/
.env
*.log
.DS_Store
EOF

cat > .gitattributes <<'EOF'
* text=auto
*.sh text eol=lf
*.bat text eol=crlf
*.png binary
*.jpg binary
EOF

cat > CHANGELOG.md <<'EOF'
Changelog

All notable changes to this project will be documented in this file.
EOF

cat > CONTRIBUTING.md <<'EOF'
Contributing

Please create a feature branch from main, submit a pull request, and wait for review before merging.
EOF

git add .
git commit -m "chore: initialize repository structure"
```

注意：上述 README 中的代码块嵌套在 shell heredoc 中时，某些 shell 或编辑器高亮可能不友好。实际项目可以直接用编辑器创建文件。

## 创建远程仓库

以 GitHub 为例，如果本地已初始化并提交，远程页面创建仓库时不要勾选 README、`.gitignore`、LICENSE。创建后执行：

```bash
git remote add origin git@github.com:org/demo-service.git
git push -u origin main
```

以 Gitee 为例：

```bash
git remote add gitee git@gitee.com:org/demo-service.git
git push -u gitee main
```

同时维护两个远程：

```bash
git remote add origin git@github.com:org/demo-service.git
git remote add gitee git@gitee.com:org/demo-service.git
git push -u origin main
git push -u gitee main
```

查看远程状态：

```bash
git remote -v
git branch -vv
```

## 仓库平台参数清单

创建仓库后，应检查以下平台参数：

| 参数 | 建议值 | 说明 |
| --- | --- | --- |
| 默认分支 | `main` | 统一主线名称 |
| 可见性 | Public/Private | 按代码属性设置 |
| Issues | 开启 | 缺陷和需求跟踪 |
| Wiki | 视情况开启 | 长文档可使用独立 docs 仓库替代 |
| Discussions | 开源项目建议开启 | 用于问答和社区讨论 |
| Projects/看板 | 团队项目建议开启 | 管理迭代和任务 |
| Pages | 文档站点开启 | 静态文档发布 |
| Webhooks | 按需开启 | 通知外部系统或部署系统 |
| Secrets | 必须严格管理 | CI/CD 凭据不得入库 |
| Branch protection | 必须配置 | 保护主线和发布分支 |
| Tag protection/rulesets | 发布项目建议配置 | 防止误删或移动版本标签 |

## 分支策略设计

小团队推荐 GitHub Flow 或 Trunk-Based Development：

```text
main
  ├── feature/login
  ├── fix/order-timeout
  └── docs/api-guide
```

规则：

- 所有功能分支从 `main` 创建。
- 所有改动通过 PR 合并。
- `main` 始终可构建、可测试、可发布。
- 发布通过标签和 Release 表达。

中大型团队或传统版本交付可使用简化 Git Flow：

```text
main
  ├── develop
  │   ├── feature/a
  │   └── feature/b
  ├── release/1.4
  └── hotfix/1.4.1
```

规则：

- `develop` 集成功能。
- `release/x.y` 只允许修复发布问题。
- `main` 保存已发布版本。
- `hotfix/x.y.z` 从 `main` 拉出，修复后合回 `main` 和 `develop`。

## 保护分支配置模板

主分支保护模板：

```text
pattern: main
require_pull_request: true
required_approvals: 2
require_code_owner_review: true
require_status_checks: [build, test, lint]
require_conversation_resolution: true
allow_force_pushes: false
allow_deletions: false
require_linear_history: optional
```

发布分支保护模板：

```text
pattern: release/*
require_pull_request: true
required_approvals: 1
allowed_pushers: release-managers
allow_force_pushes: false
allow_deletions: false
```

标签保护模板：

```text
pattern: v*
allowed_creators: maintainers
allow_deletions: false
allow_updates: false
```

这些配置在不同平台名称不完全一致，但治理目标相同：保护主线、保护发布历史、保留审查记录、防止不可逆误操作。

## 维护 README

README 不应只写一句项目介绍。可维护 README 至少包含：

| 小节 | 内容 |
| --- | --- |
| 项目简介 | 解决什么问题 |
| 架构概览 | 核心模块、依赖、部署关系 |
| 快速开始 | 安装、配置、启动、测试 |
| 配置说明 | 环境变量、配置文件、默认值 |
| 开发流程 | 分支、提交、测试、PR |
| 发布方式 | 标签、Release、部署命令 |
| 常见问题 | 本地启动、权限、依赖问题 |

## 维护 CHANGELOG

建议按版本维护变更记录：

```markdown
Changelog

### v1.1.0

### Added

- Add user profile API.

### Fixed

- Fix token refresh race condition.

### Changed

- Update default timeout from 3s to 5s.
```

CHANGELOG 应面向使用者，而不仅是提交列表。内部重构如果不影响用户，可以放在“Changed”或省略；破坏性变更必须明确迁移方式。

## 维护 Issue 与 PR 模板

GitHub 可放在 `.github/ISSUE_TEMPLATE/` 和 `.github/pull_request_template.md`。Gitee 也支持 Issue 和 Pull Request 模板能力。

PR 模板示例：

```markdown
### 变更内容

### 测试结果

### 风险与回滚

### 关联 Issue
```

Issue 模板示例：

```markdown
### 问题描述

### 复现步骤

### 期望结果

### 实际结果

### 环境信息
```

模板的作用是降低沟通成本，让审查者获得稳定上下文。
