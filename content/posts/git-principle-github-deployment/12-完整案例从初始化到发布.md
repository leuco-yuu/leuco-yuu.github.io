# 完整案例从初始化到发布

## 案例背景

假设团队要开发一个名为 `weather-api` 的服务。目标是建立一个可维护仓库，并同时托管到 GitHub 和 Gitee。项目采用 `main` 主分支、短期功能分支、Pull Request 审查、CI 检查、语义化版本和 Release 发布。

目标状态：

| 项目 | 目标 |
| --- | --- |
| 默认分支 | `main` |
| 远程平台 | GitHub 为主，Gitee 为镜像 |
| 分支策略 | GitHub Flow |
| 提交规范 | Conventional Commits |
| 发布方式 | `v*` 标签触发 Release |
| 主分支保护 | PR、审查、CI、禁止强推 |
| 文档 | README、CONTRIBUTING、CHANGELOG |

## 初始化仓库

```bash
mkdir weather-api
cd weather-api
git init -b main

mkdir -p src tests docs scripts
cat > README.md <<'EOF'
Weather API

Weather API provides a minimal HTTP API for querying weather data.

### Quick start

npm install
npm test
npm run dev

### Release

Stable versions are published as Git tags and platform releases.
EOF

cat > .gitignore <<'EOF'
node_modules/
dist/
coverage/
.env
*.log
.DS_Store
EOF

cat > .gitattributes <<'EOF'
* text=auto
*.sh text eol=lf
*.bat text eol=crlf
*.png binary
EOF

cat > CONTRIBUTING.md <<'EOF'
Contributing

Create a branch from main, use Conventional Commits, open a pull request, and wait for review.
EOF

cat > CHANGELOG.md <<'EOF'
Changelog

### Unreleased

- Initial project structure.
EOF

git add .
git commit -m "chore: initialize weather api repository"
```

## 连接 GitHub 和 Gitee

先在 GitHub 创建空仓库 `org/weather-api`，不要初始化 README、`.gitignore`、LICENSE。再在 Gitee 创建空仓库 `org/weather-api`。

```bash
git remote add origin git@github.com:org/weather-api.git
git remote add gitee git@gitee.com:org/weather-api.git

git push -u origin main
git push -u gitee main
```

检查：

```bash
git remote -v
git branch -vv
```

## 配置主分支保护

在 GitHub 上为 `main` 配置：

```text
Require a pull request before merging: enabled
Required approvals: 1
Require status checks to pass: enabled
Require conversation resolution before merging: enabled
Allow force pushes: disabled
Allow deletions: disabled
```

在 Gitee 上为 `main` 配置保护分支：

```text
禁止直接推送到 main
要求 Pull Request 合并
要求至少一名维护者评审
禁止强制推送
禁止删除分支
```

如果团队有模块负责人，增加 CODEOWNERS：

```bash
mkdir -p .github
cat > .github/CODEOWNERS <<'EOF'
* @org/core-team
/docs/ @org/docs-team
/src/providers/ @org/integration-team
EOF

git add .github/CODEOWNERS
git commit -m "chore: add code owners"
git push origin main
git push gitee main
```

主分支保护开启后，后续这类提交应通过 PR，而不是直接 push。

## 开发功能分支

```bash
git fetch origin
git switch -c feature/current-weather origin/main

cat > src/current-weather.js <<'EOF'
export function getCurrentWeather(city) {
  if (!city) {
    throw new Error('city is required')
  }
  return { city, temperature: 20, condition: 'cloudy' }
}
EOF

cat > tests/current-weather.test.js <<'EOF'
import { getCurrentWeather } from '../src/current-weather.js'

test('returns weather by city', () => {
  expect(getCurrentWeather('Helsinki').city).toBe('Helsinki')
})
EOF

git add src/current-weather.js tests/current-weather.test.js
git commit -m "feat(weather): add current weather query"
git push -u origin feature/current-weather
```

在 GitHub 创建 PR，目标为 `main`。如果同时使用 Gitee 作为协作平台，也推送同名分支：

```bash
git push -u gitee feature/current-weather
```

## 同步主线并解决冲突

PR 审查期间，如果主线前进：

```bash
git fetch origin
git switch feature/current-weather
git rebase origin/main
```

如有冲突：

```bash
git status
```

编辑冲突文件后：

```bash
git add <resolved-file>
git rebase --continue
```

因为 rebase 重写了功能分支提交，需要更新远程功能分支：

```bash
git push --force-with-lease origin feature/current-weather
```

只对个人功能分支使用 `--force-with-lease`，不要对共享主分支使用。

## 合并与清理

PR 通过审查和 CI 后合并。若平台选择 squash merge，主分支会得到一个整洁提交。合并后：

```bash
git switch main
git pull --ff-only origin main
git branch -d feature/current-weather
git fetch --prune
```

删除远程功能分支：

```bash
git push origin --delete feature/current-weather
git push gitee --delete feature/current-weather
```

同步 Gitee 主线：

```bash
git push gitee main
```

## 准备发布

更新 CHANGELOG：

```bash
cat > CHANGELOG.md <<'EOF'
Changelog

### v1.0.0

### Added

- Add current weather query API.

### Notes

- This is the first stable release.
EOF

git add CHANGELOG.md
git commit -m "docs: update changelog for v1.0.0"
git push origin main
git push gitee main
```

打标签：

```bash
git switch main
git pull --ff-only origin main
git tag -a v1.0.0 -m "Release v1.0.0"
git push origin v1.0.0
git push gitee v1.0.0
```

## 发布 GitHub Release

使用 GitHub CLI：

```bash
gh release create v1.0.0 \
  --repo org/weather-api \
  --title "v1.0.0" \
  --notes-file CHANGELOG.md
```

如果有构建产物：

```bash
npm run build
tar -czf weather-api-v1.0.0.tar.gz dist
gh release upload v1.0.0 weather-api-v1.0.0.tar.gz --repo org/weather-api
```

Release 页面说明应包含新增功能、升级方式、兼容性、校验信息和已知问题。

## 发布 Gitee Release

在 Gitee 仓库页面进入发行版，选择 `v1.0.0` 标签，填写标题 `v1.0.0`，将 CHANGELOG 中本版本内容作为发行说明，并上传同一份构建产物。保持 GitHub 与 Gitee 的版本号、说明和附件一致，避免用户下载到不同内容。

## 发布后维护

发布后执行：

```bash
git log --oneline --decorate -n 5
git tag -l "v1.*"
git status -sb
```

维护任务：

| 任务 | 说明 |
| --- | --- |
| 关闭已完成 Issue | 在 PR 或 Release 中关联 |
| 检查 Release 附件 | 下载验证可用性 |
| 同步双平台 | 分支、标签、Release 内容一致 |
| 清理临时分支 | 删除已合并功能分支 |
| 记录后续计划 | 在 Issue、Milestone 或项目看板中维护 |

如果发现严重缺陷，从 `v1.0.0` 或 `main` 创建 hotfix 分支：

```bash
git switch -c hotfix/1.0.1 v1.0.0
```

修复、测试、打 `v1.0.1` 标签并发布补丁 Release，然后把修复合回 `main`。
